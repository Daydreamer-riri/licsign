# Android TV Verification

The Android TV app should activate once, persist the returned compact JWS, then verify
that token locally on every launch.

## Machine Hash

The service expects `machine_hash` to be a lowercase SHA-256 hex digest generated by
the client. Do not send raw hardware identifiers.

For Android TV, choose identifiers carefully. A practical V1 approach is:

- Normalize a stable app-scoped install/device identifier.
- Include product/app namespace.
- Hash the normalized string with SHA-256.

Avoid identifiers that are likely to change after app updates, account changes, or
normal device restarts.

## License Token

The activation API returns:

- `license`: decoded payload for convenience.
- `signature`: detached JWS signature segment.
- `token`: compact JWS string to store.

Store `token` locally. Treat local storage as user-editable and always verify it.

## Kotlin Verification Shape

V1 uses `ES256`, which is ECDSA P-256 with SHA-256. A compact JWS is:

```text
base64url(header).base64url(payload).base64url(raw_signature)
```

Android's `Signature` API typically expects DER-encoded ECDSA signatures for
`SHA256withECDSA`, while JWS stores the raw 64-byte `r || s` signature. The Android
client should convert raw JWS ECDSA signatures to DER before calling `verify`.

Verification steps:

1. Split token into three segments.
2. Decode header and check `alg == "ES256"` and trusted `kid`.
3. Decode payload and check `product_code`, `machine_hash`, `expires_at`, `issuer`.
4. Rebuild signing input as `headerSegment + "." + payloadSegment`.
5. Convert the signature segment from raw `r || s` to DER.
6. Verify with embedded public key and `SHA256withECDSA`.

Keep the public key in app resources or code. Never ship the private key.

## Trial Tokens

The same verification flow works for trial tokens issued by
`POST /api/client/trial`. Two payload fields differ:

- `kind == "trial"` (paid activation tokens omit `kind` or set it to `"license"`)
- `license_id == null`

The signature, header, `machine_hash`, `expires_at`, and `issuer` checks are
identical. Clients can use `kind` to drive UI affordances such as a "Trial,
expires in N days" badge, or to decide whether to surface a purchase prompt
when the token nears expiry.

Trial tokens carry a short independent TTL (configured per product). When a
trial token is about to expire, the client should call
`POST /api/client/trial` again with the same `product_code + machine_hash`:

- If the product's trial window is still open, the server returns a fresh token.
- If the window has closed, the server returns `TRIAL_INACTIVE` and the client
  should prompt the user to enter a paid activation code.
